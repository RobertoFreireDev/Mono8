# Task 002: Add sprite flags UI to the sprite editor

## Goal
Add a row of 8 toggleable flag indicators to the sprite editor so each sprite's flag byte can be edited visually, wired to the existing flag functions in `src/core/sprites/SpriteSheet.cs`.

## Current state
- `SpriteSheet.cs` already stores and persists per-sprite flags:
  - `byte[] Flags` (one byte per sprite, 8 bits).
  - `GetFlags(spriteId)`, `GetFlag(spriteId, flag)`, `SetFlag(spriteId, flag, value)`, `SetFlags(spriteId, value)`.
  - `LoadFlags`/`ToFlagLines` handle serialization to/from the `.flags` save file (see `src/Mono8API.cs:27,36`).
- `Mono8API.cs` already exposes flags to game scripts: `fget(spriteId)`, `fget(spriteId, flag)`, `fset(spriteId, flag, value)`, `fset(spriteId, value)` (`src/Mono8API.cs:250-256`).
- `MapSheet.cs:98` already consumes sprite flags for layer filtering (`GetFlags(tileIndex) & layerMax`), so flag data is meaningful beyond the editor.
- `src/editor/SpriteEditor.cs` has no UI for flags today — there's no way for a user to set them from the editor. The currently selected sprite is tracked in `sprNmbr`.
- The editor already has a similar small-button-row pattern to copy from: `pageButtons` (page selector, `SpriteEditor.cs:39,73-78,268-275,424-435`) and `toolButtons`.

## Changes required
1. In `SpriteEditor.cs`, add 8 small clickable flag indicators (one per bit) for the currently selected sprite (`sprNmbr`):
   - Lay them out similarly to `pageButtons`/`toolButtons` (e.g. a row of `Constants.GameDataSizes.TileSize`-sized squares) in an unused area of the editor UI (e.g. near `sprNmbrLabelArea` or below the palette/tool row).
   - Each indicator should render differently when the corresponding bit is set vs. unset (e.g. filled vs. outline, similar to how `pageButtons` swap `PageIconSelected`/`PageIconNotSelected` and use `_api.pal(...)` to dim unselected icons).
2. Wire clicks to `Mono8API.SpriteSheet.SetFlag(sprNmbr, flagIndex, !currentValue)`, reading current state via `Mono8API.SpriteSheet.GetFlag(sprNmbr, flagIndex)`.
3. Update the `Draw()` method to render the 8 indicators for `sprNmbr` every frame, reflecting the live flag state (including when the selected sprite changes via the sprite viewer or page switch).
4. Handle mouse input for the new flag row in `Update()` following the existing pattern (check bounds, `_api.mouselp()` for a single click-toggle, `break`/continue logic consistent with `toolButtons`/`pageButtons`).
5. Make sure flag edits participate in undo/redo if expected — check whether `SetFlag` should call `SaveSnapshot()` similar to pixel-editing operations, or whether flags are intentionally excluded from the undo stack (currently `SetFlag`/`SetFlags` do not call `SaveSnapshot()`). Decide and document behavior; if flags should be undoable, add snapshot support without breaking the existing `Data`-only undo stack (flags may need their own undo/redo, or a combined snapshot).

## Acceptance criteria
- Sprite editor shows 8 flag toggles for the currently selected sprite.
- Clicking a toggle flips that bit via `SpriteSheet.SetFlag` and is immediately reflected in the UI.
- Toggle state persists correctly across sprite/page switches and through save/load (`ToFlagLines`/`LoadFlags`).
- No regression to existing tools, page selector, or undo/redo behavior.
