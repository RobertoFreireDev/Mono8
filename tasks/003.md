# Task 003: Shared editor menu bar with icon-based editor switching (PICO-8 style)

## Goal
Add a menu bar that is shared across all editors (not owned by `SpriteEditor` alone), showing a row of icons at the top of the screen that let the user switch between editors â€” like the sprite/map/sfx/music editor icons in the top-right corner of PICO-8. This must work today with only `SpriteEditor` existing, but be built so additional editors (map editor, sfx editor, music editor, etc.) can register into the same bar later without rework.

## Current state
- `src/editor/IEditor.cs` defines the `IEditor` contract (`Init`, `Update`, `Draw`). `SpriteEditor` is currently the only implementation.
- `src/Mono8API.cs:5,15`: `Mono8API.editor` is a single hardcoded `IEditor` field, set once in the constructor (`editor = new SpriteEditor(this)`). There is no concept of multiple editors or switching between them.
- `Mono8API.Update` (`Mono8API.cs:40-59`) calls `editor.Init()` once and `editor.Update(...)` every frame (skipped while `Menu.IsPaused()`); the draw call presumably lives wherever `Mono8API.Draw` is (not shown here but follows the same single-`editor` pattern).
- `SpriteEditor.Draw()` (`SpriteEditor.cs:329`) currently paints its own top strip itself: `_api.rectfill(0,0,Constants.Screen.ResolutionX,Constants.GameDataSizes.TileSize,Constants.Colors.Orange)`. This is editor-specific chrome, not a shared bar, and leaves no room for editor-switch icons â€” a new shared bar needs to either replace this strip or coexist with it without overlapping the sprite number label area (`sprNmbrLabelArea`, `SpriteEditor.cs:41,71`) or existing page buttons (`pageButtons`, right-aligned at `Constants.Screen.ResolutionX - SpritePages * size`).
- Icon-button prior art already exists and should be reused/extended rather than reinvented:
  - `src/editor/Button.cs`: simple clickable icon button (`Bounds`, `IconIndex`, `IsClicked`, `Draw` with `pal()` dimming when unselected).
  - `SpriteEditor.cs:39,73-78,268-275,424-435` (`pageButtons`): a manually laid out row of icon rectangles with click handling and selected/unselected icon swap (`PageIconSelected = 45`, `PageIconNotSelected = 46`).
- `src/core/icons/IconSheet.cs`: icon sheet is `IconSheetX=256, IconSheetY=16` â†’ 32 columns x 2 rows = 64 icon slots (`Constants.cs:35-36`), drawn via `IconSheet.Draw(n, x, y)` / exposed to editors as `_api.icon(...)`. Icons `45`/`46` are already used for the sprite page selector; a new editor-switch icon set needs its own unused indices.
- `src/core/graphics/Menu.cs` is an unrelated pause menu (`Continue`/`Restart`/`Exit` overlay triggered by Start button) â€” do not confuse this with the new editor menu bar; pick a distinct name (e.g. `EditorMenuBar` or `EditorSwitcher`) to avoid confusion with `core.graphics.Menu`.
- `Constants.Screen.ResolutionX = 256`, `ResolutionY = 144`, `TileSize = 8` (`Constants.cs:27-28,39`).

## Changes required
1. Introduce a small editor-registry concept in place of the single hardcoded `Mono8API.editor` field:
   - Keep an ordered list/array of available `IEditor` instances (initially just `SpriteEditor`), each associated with an icon index and a display label.
   - Track which editor is currently active; expose it the same way `Mono8API.editor` is used today (`editor.Init()/Update()/Draw()` call sites in `Mono8API.cs` and anywhere else `Mono8API.editor` is referenced) so existing call sites keep working, or update those call sites to go through the new registry/active-editor accessor.
   - Switching editors should call `Init()` on the newly activated editor (mirroring how `Mono8API.Update` currently calls `editor.Init()` once via the `firstTime` flag) â€” decide whether `Init()` should re-run every time an editor becomes active or only the first time, and keep it consistent with existing sprite-editor reset behavior (`Menu.cs:81`, `Mono8API.editor.Init()` on "Restart").
2. Build a shared, reusable menu bar component (new file, e.g. `src/editor/EditorMenuBar.cs`) that:
   - Draws a fixed-height strip (reuse `Constants.GameDataSizes.TileSize` height, matching the strip `SpriteEditor.Draw()` currently paints at y=0) spanning the top of the screen.
   - Renders one icon button per registered editor, right- or left-aligned similar to the existing `pageButtons` layout pattern, using distinct new icon indices from the icon sheet (avoid colliding with `45`/`46` and any other in-use indices): sprite editor icon = `15`, map editor icon = `16` â€” reuse the `Button` class or the same selected/unselected `pal()`-dimming convention used by `toolButtons`/`pageButtons`.
   - Handles mouse clicks on each icon (same `Rectangle.Contains(mouse.x, mouse.y)` + `_api.mouselp()` pattern used throughout `SpriteEditor.Update`) and switches the active editor via the registry from step 1.
   - Is driven from `Mono8API` (or wherever the top-level `Update`/`Draw` loop lives) so it runs and draws every frame regardless of which editor is active, rather than being embedded inside `SpriteEditor` itself.
3. Update `SpriteEditor` so it no longer paints its own top orange strip as ad-hoc chrome that conflicts with the new bar â€” either remove that `rectfill` call and let the shared bar own the top row, or shrink/reposition sprite-editor content so the shared bar sits cleanly above/alongside it without visual overlap. Re-check that `sprvwrarea`, `sprNmbrLabelArea`, and `pageButtons` positions (all currently derived from `Constants.Screen.ResolutionY`/`labelRowY`) still fit and don't collide with the new bar.
4. Make sure input routing doesn't double-handle clicks: when the mouse is over the new menu bar, `SpriteEditor.Update` (and any future editor's `Update`) should not also interpret that click as canvas/tool/page input â€” follow the existing pattern where `SpriteEditor.Update` already gates behavior by `sprvwrarea`/`sprcnvsarea`/`palettearea.Contains(...)` checks (`SpriteEditor.cs:201,212,248`) with a final `else` for buttons; the menu bar's screen region needs to be excluded the same way, or checked/consumed before delegating to the active editor.
5. Add at least the icon(s) needed for the sprite editor (index `15`) and map editor (index `16`) into the icon sheet data/asset the game loads (`FileIO.Read(..., Constants.File.Extensions.IconSheet, ...)`, `IconSheet.LoadIcons`) so the new button(s) actually render something meaningful instead of a blank/garbage tile.

## Acceptance criteria
- A menu bar is visible at the top of the screen in every editor (currently just the sprite editor), independent of which specific `IEditor` is active.
- The bar shows one icon per registered editor; clicking an icon switches the active editor and that editor's `Update`/`Draw` take over the main content area.
- Adding a future editor (e.g. a map editor) only requires implementing `IEditor` and registering it (icon + instance) â€” no changes needed inside `SpriteEditor` or duplication of bar/layout code.
- No regression to existing sprite editor functionality: tool selection, page selector, palette, canvas drawing, undo/redo, copy/paste, flip/rotate all continue to work, and none of their click regions overlap with or are obscured by the new menu bar.
- No hardcoded assumption remains that `SpriteEditor` is the only editor (e.g. `Mono8API.editor` being a single fixed instance with no way to switch).
