# Task 001: Add optional transparency parameter to drawing functions

## Goal
Add an optional int `transparency` parameter (clamped 0–10, where 0 = fully
opaque and 10 = fully transparent) to `pixel`, `line`, `rect`, `rectfill`,
`circ`, `circfill`, `oval`, `ovalfill`, `spr`, `sspr`, and `map`, so scripts
can fade drawn pixels/sprites/tiles without pre-blending colors. Also remove
`map`'s existing `color` parameter, since it is accepted by `MapSheet.DrawMap`
but never used.

## Current state
- Interface signatures live in `src/IMono8API.cs`:
  - `pixel(x, y, color)` (25), `line(x0,y0,x1,y1,color)` (27),
    `rect(x0,y0,x1,y1,color)` (29), `rectfill(x0,y0,x1,y1,color)` (31),
    `circ(x,y,radius,color)` (33), `circfill(x,y,radius,color)` (35),
    `oval(x0,y0,x1,y1,color)` (37), `ovalfill(x0,y0,x1,y1,color)` (39),
    `spr(spriteId,x,y,width=1,height=1,scale=1,flipX=false,flipY=false)`
    (41-49), `sspr(sx,sy,sw,sh,dx,dy,dw=-1,dh=-1,flipX=false,flipY=false)`
    (51-55).
  - `map(cellX,cellY,screenX,screenY,cellWidth=40,cellHeight=23,layerMax=0,
    color=-1)` (90-96).
- Implementations in `src/Mono8API.cs` (140-213, 250-253) delegate straight
  through to the drawing backends with no transparency concept.
- Low-level primitives in `src/core/graphics/PixelledSpriteBatch.cs`
  (`DrawPixel` 36, `DrawRect` 41, `DrawRectFill` 65, `DrawLine` 81, `DrawCirc`
  96, `DrawCircFill` 110, `DrawOval` 140, `DrawOvalFill` 145) all take a
  single `int colorIndex` and call `ColorPalette.GetColor(colorIndex)`
  (`ColorPalette.cs:79-97`), which always returns a fully-opaque `Color`.
- `spr`/`sspr` are backed by `SpriteSheet.Draw`/`DrawSub`
  (`src/core/sprites/SpriteSheet.cs:513-554`), which loop over the 16
  palette indices and call `mono8.SpriteBatch.Draw(texture, destination,
  source, effects, ci)` — this overload is
  `PixelledSpriteBatch.Draw(Texture2D, Rectangle, Rectangle, SpriteEffects,
  int colorId)` (`PixelledSpriteBatch.cs:70-74`), which also resolves color
  via `ColorPalette.GetColor(colorId)`.
- `map` is backed by `MapSheet.DrawMap` (`src/core/maps/MapSheet.cs:78-106`).
  Its final parameter, `int color`, is threaded through from
  `Mono8API.map()` (`Mono8API.cs:250-253`) but is **never read inside
  `DrawMap`** — every tile renders via
  `Mono8API.SpriteSheet.Draw(tileIndex, px+…, py+…)` (100-103) with no color
  argument passed at all. It is dead code today.
- `ColorPalette.cs` already has a separate, unrelated binary transparency
  system: `_paltFlags`/`IsDrawTransparent(colorIndex)`/`Palt(...)`, driving
  the `palt()` API. That mechanism skips a palette index entirely (used only
  by `spr`/`sspr` today) and is orthogonal to the new int transparency
  parameter — do not conflate the two.
- There is no existing fractional alpha/opacity concept anywhere in the
  codebase; `Color` values produced by `ColorPalette.GetColor` are always
  fully opaque (alpha 255) except the special out-of-range `TransparentColor`
  (`ColorPalette.cs:6`).

## Changes required
1. In `ColorPalette.cs`, add an alpha-aware color resolution path, e.g. an
   overload `GetColor(int id, int transparency)` that clamps `transparency`
   to `[0, 10]`, resolves the base `Color` the same way `GetColor(int id)`
   does, and returns it with alpha scaled down as `transparency` increases
   (`0` → alpha 255/fully opaque, `10` → alpha 0/fully invisible). Keep
   `GetColor(int id)` as-is (or have it delegate to the new overload with
   `transparency = 0`) so unrelated callers are unaffected.
2. Add `int transparency = 0` as the last parameter on `pixel`, `line`,
   `rect`, `rectfill`, `circ`, `circfill`, `oval`, `ovalfill` in
   `IMono8API.cs` and their implementations in `Mono8API.cs`, threading the
   value down into `PixelledSpriteBatch`'s corresponding `Draw*` methods.
3. In `PixelledSpriteBatch.cs`, add `int transparency = 0` to `DrawPixel`,
   `DrawRect`, `DrawRectFill`, `DrawLine`, `DrawCirc`, `DrawCircFill`,
   `DrawOval`, `DrawOvalFill`, and any internal calls between them (e.g.
   `DrawRect`/`DrawCirc`/`DrawOval` calling `DrawRectFill`/`DrawPixel`
   internally), replacing `ColorPalette.GetColor(colorIndex)` calls with
   `ColorPalette.GetColor(colorIndex, transparency)`.
4. Add `int transparency = 0` to `spr` and `sspr` in `IMono8API.cs`/
   `Mono8API.cs`, threading it into `SpriteSheet.Draw`/`DrawSub`
   (`SpriteSheet.cs:513-554`), which pass it into the
   `mono8.SpriteBatch.Draw(texture, destination, source, effects, ci)` call
   — extend that `PixelledSpriteBatch.Draw` overload
   (`PixelledSpriteBatch.cs:70-74`) to accept `transparency` and use
   `ColorPalette.GetColor(colorId, transparency)`.
5. In `IMono8API.cs`/`Mono8API.cs`, change `map`'s signature from
   `map(cellX, cellY, screenX, screenY, cellWidth = 40, cellHeight = 23,
   layerMax = 0, color = -1)` to drop `color` and add
   `int transparency = 0` in its place (or appended after `layerMax`,
   whichever reads more naturally as the final optional parameter). Update
   `MapSheet.DrawMap` (`MapSheet.cs:78-106`) to drop its unused `color`
   parameter and accept `transparency`, passing it into the
   `SpriteSheet.Draw(tileIndex, px, py, ...)` call — which requires
   extending `SpriteSheet.Draw` to accept a `transparency` argument (shared
   with change 4).
6. Ensure `transparency` is clamped to `[0, 10]` in exactly one place (the
   `ColorPalette` overload from step 1 is the natural spot) rather than
   re-validated at every call site.

## Acceptance criteria
- `pixel`, `line`, `rect`, `rectfill`, `circ`, `circfill`, `oval`,
  `ovalfill`, `spr`, `sspr`, and `map` each accept a trailing optional
  `int transparency = 0` parameter.
- Passing `transparency` values outside `[0, 10]` clamp into range rather
  than throwing or drawing incorrectly (e.g. negative values behave as `0`,
  values above 10 behave as `10`).
- Calling any of these functions without the new parameter renders
  identically to current (fully opaque) behavior — no regression for
  existing scripts that don't pass `transparency`.
- `map`'s `color` parameter is removed from `IMono8API.cs`, `Mono8API.cs`,
  and `MapSheet.DrawMap`; no dead/unused parameter remains.
- `spr`/`sspr`'s existing `palt()`-driven binary transparency
  (`ColorPalette.IsDrawTransparent`) continues to work unchanged and
  independently of the new int `transparency` parameter.
